# 中外法学智能编审系统 — AI 融入建议

本文档基于当前代码与 PRD，整理**已实现的 AI 能力**与**可进一步融入的 AI 方向**，便于按优先级迭代。

---

## 一、当前已实现的 AI 能力

| 能力 | 位置 | 说明 |
|------|------|------|
| **大模型调用** | `backend/app/services/llm.py` | 阿里云百炼（OpenAI 兼容），`chat_completion()`，支持 system/user 消息 |
| **AI 初审报告** | `backend/app/services/review_service.py` | 基于解析后的标题/摘要/正文/脚注/结构，生成 Markdown 格式初审报告（概览、结构评估、形式规范、核心优点与问题、修改意见） |
| **编辑端触发报告** | 编辑工作台、稿件详情页 | 按钮「生成 AI 初审报告」，调用 `POST editor/manuscripts/{id}/ai-review` |
| **AI 对话 API** | `backend/app/api/v1/ai.py` | `GET /ai/status`、`POST /ai/chat`，仅编辑可调用；**前端未提供对话 UI** |

---

## 二、PRD 规划但尚未实现的能力

- **引注基础检查**：规则引擎 + 问题列表 → 表 `citation_issues` 已存在，接口目前返回空数组。
- **站内相似度检测**：向量检索 + 相似段落标注 → 表 `similarity_results`、`knowledge_chunks` 已存在，pgvector 未接入，无切块/向量化/检索。
- **知识库入库与向量化**：投稿后自动切块、打元数据、向量化写入 → 未实现。
- **AI 风险提示**：模板化语言、论证不足、引注密度异常等 → 目前合并在初审报告文本中，无独立结构化输出。

---

## 三、可融入 AI 的方向（按推荐优先级）

### 1. 编辑端「基于当前稿件的智能对话」（高价值、低成本）

- **现状**：`/ai/chat` 已存在，前端未使用。
- **建议**：在稿件详情页增加「AI 助手」入口（侧边栏或弹窗），对话时自动把当前稿件摘要、标题、关键词（或报告摘要）注入 system/首条 user，便于编辑问：
  - 「这篇更适合哪个栏目？」
  - 「根据报告给一段退修意见草稿」
  - 「概括一下主要问题」
- **实现要点**：前端调用 `aiApi.chat`，首条消息由前端或后端拼接「当前稿件上下文」+ 用户问题；可选在后端提供 `POST /editor/manuscripts/{id}/ai-chat`，由后端注入上下文，避免前端传长文本。

### 2. 退修意见智能生成（高价值、中成本）

- **场景**：编辑点击「退修」时，可一键生成退修意见草稿，再人工修改后提交。
- **实现**：在 `revision_request` 时增加可选参数 `generate_comment_with_ai: true`，后端用当前稿件的 AI 初审报告 + 退修模板，调用 LLM 生成一段退修意见，返回给前端填入意见框；编辑可修改后再提交。
- **依赖**：已有初审报告与退修模板，仅需新接口或扩展现有 action 接口。

### 3. 引注规则检查（规则引擎 + 可选 LLM 补充）（中高价值、中成本）

- **目标**：从 `manuscript_parsed.footnotes_raw` / `references_raw` 中检测格式不统一、法条书写不规范、缺页码等，写入 `citation_issues`。
- **实现思路**：
  - **先规则**：正则 + 简单规则（如脚注序号连续性、常见法条格式、中英文文献必备字段），输出 `location`、`issue_type`、`description`、`suggestion`、`severity`。
  - **可选 LLM**：对难以用规则覆盖的段落，用 LLM 做「这段脚注是否符合本刊规范」的简短判断，再转为 `CitationIssue`。
- **对接**：在「生成 AI 初审报告」流程中或单独「引注检查」步骤中写入 `citation_issues`，编辑端详情页已有 `citation_issues` 占位，只需后端填入数据即可展示。

### 4. 站内相似度检测（向量检索）（高价值、成本较高）

- **目标**：投稿解析后对正文切块、向量化，与库内已有稿件/历史投稿做相似检索，将相似段落写入 `similarity_results`，并在报告中展示「近似段落」。
- **实现步骤**：
  1. **向量库**：启用 pgvector（或使用 SQLite + 本地向量文件，需额外实现检索），在 `knowledge_chunks` 增加 `embedding` 列或对接独立向量表。
  2. **Embedding 接口**：使用百炼 embedding 接口（或同一账号下兼容的 embedding API）对切块文本向量化。
  3. **入库**：解析完成后对正文按段落/固定长度切块，写入 `knowledge_chunks` 并写入向量；排除当前稿件版本，避免自匹配。
  4. **检索**：生成报告或打开详情时，对当前版本切块向量检索，取 top-k 相似块，若相似度超过阈值则写入 `similarity_results`（含 source_excerpt、target_excerpt、score、target_id）。
- **注意**：当前为 SQLite，若暂不引入 PostgreSQL，可考虑用本地向量库（如 Chroma、Faiss）或先做「关键词/标题相似」的轻量版，再迁 pgvector。

### 5. AI 风险提示结构化（中价值、中低成本）

- **目标**：在初审报告中单独列出「AI 风险提示」项（如：语言模板化、论证支撑不足、引注密度异常、概念堆砌、段落结构过于均匀），便于编辑快速扫一眼。
- **实现**：
  - **方案 A**：在现有 `generate_full_ai_report` 的 prompt 中要求模型输出固定 Markdown 小节「## AI 风险提示」，并在解析报告时提取该小节，以结构化字段（如 `report.content.ai_risks: string[]`）存到 `content` JSON 中，前端单独展示。
  - **方案 B**：单独调用一次 LLM，仅针对「AI 风险」做短文本分类/列表生成，写入 `content.ai_risks`。
- **推荐**：先采用方案 A，不增加调用次数，仅规范输出格式与解析逻辑。

### 6. 作者端投稿前自检（可选、轻量）（中价值、低成本）

- **场景**：作者在投稿页或「保存草稿」时，可选「规范性自检」：对标题、摘要、关键词、脚注样本做简短检查，返回提示（如「摘要建议 200–300 字」「关键词建议 3–5 个」）。
- **实现**：作者端调用一个轻量接口（如 `POST /author/manuscripts/check`），传入标题/摘要/关键词/可选的一小段脚注文本），后端用规则 + 可选一次短 LLM 调用，返回 `{ suggestions: string[] }`，前端以非阻塞方式展示。
- **注意**：需控制频率与 token，避免滥用；可仅对已登录作者开放并做限流。

### 7. 初审报告结构化输出（便于展示与引用）（中价值、中成本）

- **现状**：报告为整段 Markdown 存在 `content.text`，前端整体渲染。
- **建议**：让 LLM 按固定 JSON 结构输出（如 `overview`、`structure_assessment`、`citation_issues_summary`、`similarity_risks`、`ai_risks`、`revision_suggestions`），或先 Markdown 再在后端用规则/小模型解析为 JSON，存入 `content`。这样可：
  - 在编辑驾驶舱分块展示（如「核心问题」卡片、「修改意见」列表）；
  - 退修意见生成时直接引用「修改意见」条目；
  - 与 `citation_issues`、`similarity_results` 表结构对齐，便于后续打通。

### 8. 知识库与「相关文献」推荐（中长期）

- **目标**：在编辑查看稿件时，展示「与本文相关的已发表/历史投稿」列表，辅助选题与重复度判断。
- **实现**：依赖「站内相似度」的向量检索；在检索时区分 `source_type`（如 manuscript / published），对 published 单独排序展示为「相关文献」。初版可在相似度结果中增加「来源类型」展示，不单独做推荐接口。

---

## 四、建议实施顺序（短期）

1. **编辑端基于稿件的 AI 对话**：前端增加对话入口，后端可选提供带上下文注入的 `ai-chat`，快速提升编辑体验。
2. **退修意见智能生成**：在退修流程中增加「AI 生成草稿」按钮或参数，复用现有报告与模板。
3. **AI 风险提示结构化**：在现有报告 prompt 中约定「AI 风险提示」小节并解析到 `content.ai_risks`，前端单独展示。
4. **引注规则检查**：先实现规则引擎版，写入 `citation_issues`，编辑端展示；再视需要加 LLM 补充。
5. **站内相似度**：在具备 pgvector 或替代向量方案后，实现切块、向量化、检索与 `similarity_results` 写入。

以上顺序可与「先业务后智能、先闭环后增强」的工程原则一致，按需分步落地。
